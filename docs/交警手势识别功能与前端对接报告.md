# 交警手势识别功能与前端对接报告

## 1. 功能概述

交警手势识别模块用于识别中国标准交警指挥手势，并在 Web 页面中展示识别结果。系统支持两类使用方式：

- 上传本地视频文件，后台按连续帧进行时序动作分析，前端在视频播放过程中叠加当前动作标签。
- 打开浏览器摄像头，前端周期性截取实时画面发送到后端，后端保持 LSTM 时序状态并返回当前手势结果。

模块覆盖的手势类别包括：

| 类别 ID | 手势名称 |
| --- | --- |
| 0 | 无手势 |
| 1 | 停止 |
| 2 | 直行 |
| 3 | 左转 |
| 4 | 左转待转 |
| 5 | 右转 |
| 6 | 变道 |
| 7 | 减速慢行 |
| 8 | 靠边停车 |

本模块前端页面位于：

```text
frontend/src/pages/PoliceGesture.tsx
```

前后端 API 封装位于：

```text
frontend/src/api/index.ts
```

后端核心实现位于：

```text
backend/server.py
```

## 2. 技术路线

系统采用“人体关键点检测 + 时序动作分类”的方案。

整体流程如下：

```text
图片 / 视频 / 摄像头画面
    ↓
OpenCV / PIL 解码图像
    ↓
等比例缩放并补边到 512x512
    ↓
Pose Estimation 提取人体关键点
    ↓
BoneLengthAngle 提取骨骼长度和角度特征
    ↓
LSTM 根据连续帧特征进行动作分类
    ↓
输出手势类别、置信度、Top-5、时间线等结果
```

模型来源为 `ctpgr-pytorch`，后端启动时加载：

```text
backend/ctpgr/checkpoints/pose_model.pt
backend/ctpgr/checkpoints/lstm.pt
```

对于视频和实时摄像头，本模块不会把每一帧独立识别后直接作为最终结果，而是保留 LSTM 的隐藏状态，使模型能够利用连续动作信息。这一点对交警手势识别很重要，因为标准交警手势通常不是单帧姿态，而是一段动作序列。

## 3. 后端接口设计

### 3.1 健康检查接口

```http
GET /api/health
```

用途：

- 检查后端服务是否启动。
- 查看模型类别数。
- 查看当前推理设备，例如 `cpu` 或 `cuda`。
- 查看 CUDA、GPU 信息。

典型返回：

```json
{
  "status": "ok",
  "model": "ctpgr-pytorch (Pose + LSTM)",
  "classes": 9,
  "device_mode": "cuda",
  "cuda_available": true,
  "device": "cuda",
  "gpu_name": "NVIDIA GeForce RTX 4070 Laptop GPU"
}
```

### 3.2 普通识别接口

```http
POST /api/police-gesture/recognize
Content-Type: multipart/form-data
```

请求参数：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| file | File | 上传的图片或视频文件 |

该接口支持图片和视频：

- 图片：执行单帧识别。
- 视频：逐帧采样，保留 LSTM 状态，最后返回完整分析结果。

视频返回的核心字段：

| 字段 | 说明 |
| --- | --- |
| gesture | 视频综合识别结果 |
| gestureId | 视频综合类别 ID |
| confidence | 综合置信度 |
| top5 | Top-5 候选类别 |
| inference_ms | 推理耗时 |
| video_duration | 视频时长 |
| video_fps | 原视频帧率 |
| sample_fps | 实际采样帧率 |
| frames | 采样帧级识别结果 |
| segments | 合并后的连续动作时间段 |

### 3.3 流式视频识别接口

```http
POST /api/police-gesture/recognize/stream
Content-Type: multipart/form-data
```

该接口用于上传视频的边分析边返回。后端使用 `StreamingResponse` 返回 `text/event-stream`，前端可以在完整视频分析结束前先拿到采样帧结果。

事件类型：

| 事件 | 说明 |
| --- | --- |
| meta | 返回视频总帧数、时长、帧率、采样间隔等元信息 |
| frame | 返回某个采样帧的识别结果和当前进度 |
| done | 返回最终平滑后的完整结果和动作时间线 |
| error | 返回异常信息 |

`frame` 事件示例：

```json
{
  "frame": 120,
  "time": 4.0,
  "gesture": "停止",
  "gestureId": 1,
  "confidence": 0.82,
  "progress": 35.4
}
```

### 3.4 浏览器兼容预览接口

```http
POST /api/police-gesture/preview
Content-Type: multipart/form-data
```

该接口用于解决部分视频“后端能分析，但浏览器不能播放”的问题。

有些视频虽然是 `.mp4`、`.mov` 或 `.mkv` 文件，但内部编码可能不是浏览器支持的 H.264/yuv420p。后端通过 FFmpeg 将上传视频转换为浏览器兼容的：

```text
MP4 + H.264 + yuv420p
```

注意：

- 转码后的视频只用于前端播放预览。
- 手势识别仍然使用用户上传的原始视频。
- 这样可以避免转码对模型输入产生额外影响。

如果系统没有安装 `ffmpeg`，后端会尝试使用 Python 依赖 `imageio-ffmpeg` 提供的内置 FFmpeg。

### 3.5 实时摄像头帧识别接口

```http
POST /api/police-gesture/stream/frame
Content-Type: multipart/form-data
```

请求参数：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| file | Blob/File | 前端从摄像头截取的 JPEG 帧 |
| stream_id | string | 实时流 ID，用于区分不同摄像头会话 |

后端处理方式：

1. 解析前端上传的 JPEG 图片。
2. 转为 OpenCV BGR 格式。
3. 等比例缩放到 `512x512`。
4. 根据 `stream_id` 获取已有 LSTM 隐藏状态。
5. 执行当前帧推理。
6. 更新该 `stream_id` 对应的 LSTM 状态。
7. 返回当前手势、置信度和 Top-5。

### 3.6 实时流状态重置接口

```http
POST /api/police-gesture/stream/reset
Content-Type: multipart/form-data
```

请求参数：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| stream_id | string | 要重置的实时流 ID |

用途：

- 开启摄像头前清空历史 LSTM 状态。
- 停止摄像头后释放该实时流的状态。
- 避免上一次实时识别的动作状态影响下一次识别。

## 4. 前端对接实现

### 4.1 API 封装

前端统一在 `frontend/src/api/index.ts` 中封装交警手势相关接口：

| 函数 | 对应后端接口 | 用途 |
| --- | --- | --- |
| `uploadPoliceGestureVideo` | `/police-gesture/recognize` | 普通图片/视频识别 |
| `streamPoliceGestureVideo` | `/police-gesture/recognize/stream` | 上传视频边分析边返回 |
| `createPoliceGesturePreview` | `/police-gesture/preview` | 生成浏览器兼容 MP4 预览 |
| `recognizePoliceGestureFrame` | `/police-gesture/stream/frame` | 实时摄像头截帧识别 |
| `resetPoliceGestureStream` | `/police-gesture/stream/reset` | 重置实时流 LSTM 状态 |

其中视频上传页面主要使用：

```text
createPoliceGesturePreview
streamPoliceGestureVideo
```

实时摄像头功能主要使用：

```text
recognizePoliceGestureFrame
resetPoliceGestureStream
```

### 4.2 上传视频识别流程

用户选择视频后，前端执行以下流程：

1. 生成本地 `Object URL`，让视频尽快出现在页面上。
2. 请求 `/police-gesture/preview`，后端自动转码生成浏览器兼容 MP4。
3. 同时启动 `/police-gesture/recognize/stream`，后端边分析边返回结果。
4. 前端收到 `meta` 事件后更新视频时长、帧率、采样帧率。
5. 前端收到 `frame` 事件后更新进度和采样帧识别结果。
6. 前端收到 `done` 事件后更新最终结果、动作时间线和动作分布。
7. 视频播放时根据当前播放时间匹配 `segments` 或最近采样帧，在视频上叠加当前动作标签。

页面展示内容包括：

- 上传视频预览。
- 视频上方动作标签叠加。
- 当前动作段时间范围。
- 分析进度条。
- 当前播放位置识别结果。
- 识别到的动作段数量。
- 主要动作。
- 动作分布。
- 视频动作时间线。

### 4.3 视频动作时间线

后端会把连续的同类动作合并为 `segments`：

```json
{
  "start": 2.4,
  "end": 5.8,
  "gesture": "直行",
  "gestureId": 2
}
```

前端使用这些时间段生成可点击时间线：

- 每个动作段用不同颜色展示。
- 点击动作段可以跳转到对应时间播放。
- 播放视频时，前端根据当前播放时间显示对应动作标签。

相比只展示一个全片动作，这种方式更适合交警手势视频，因为一个视频中可能包含多个连续指挥动作。

### 4.4 视频识别结果区域优化

当前前端不再把一个大号动作标签作为整段视频的唯一结论，而是改为多动作视频视角：

- 显示当前播放位置对应的动作。
- 显示当前时间和动作段范围。
- 显示当前帧置信度。
- 汇总动作段数量、主要动作和分析信息。
- 按总时长统计动作分布。
- 点击动作分布项可跳转到该动作首次出现位置。

这样可以避免“一个视频里有多个动作，但页面只显示一个大动作标签”造成的理解歧义。

### 4.5 实时摄像头识别流程

实时摄像头功能位于同一页面中，用户点击“实时摄像头”后：

1. 前端调用 `navigator.mediaDevices.getUserMedia` 打开摄像头。
2. 调用 `/police-gesture/stream/reset` 重置该会话的 LSTM 状态。
3. 页面用 `<video>` 实时展示摄像头画面。
4. 前端定时将当前画面绘制到 `<canvas>`。
5. 通过 `canvas.toBlob` 生成 JPEG 帧。
6. 调用 `/police-gesture/stream/frame` 上传当前帧。
7. 后端根据 `stream_id` 维护 LSTM 隐藏状态，返回当前手势。
8. 前端在实时画面上展示手势标签、置信度和候选结果。

当前前端采样间隔为：

```text
1200ms
```

因此它属于“实时摄像头截帧识别”，不是 WebRTC/RTSP 级别的连续流媒体推送。对于课程项目中的“支持实时视频流处理，Web 端展示识别结果”，该实现可以满足基本展示要求。

## 5. 前后端数据流

### 5.1 上传视频模式

```text
用户选择视频
    ↓
前端生成本地预览 URL
    ↓
前端请求 /preview 获取兼容 MP4
    ↓
前端请求 /recognize/stream 启动流式分析
    ↓
后端读取原始视频并按采样间隔推理
    ↓
后端持续返回 frame 事件
    ↓
前端更新进度、当前动作和帧结果
    ↓
后端返回 done 事件
    ↓
前端展示动作时间线和动作分布
```

### 5.2 实时摄像头模式

```text
用户点击实时摄像头
    ↓
前端打开浏览器摄像头
    ↓
前端 reset 当前 stream_id
    ↓
前端每隔约 1200ms 截取一帧
    ↓
后端根据 stream_id 读取 LSTM 状态
    ↓
后端识别当前帧并更新 LSTM 状态
    ↓
前端展示当前动作标签和置信度
```

预期结果：

## 7. 配置说明

### 7.1 GPU/CPU 推理配置

后端支持通过环境变量控制设备：

```env
CARMATE_DEVICE=auto
```

可选值：

| 值 | 说明 |
| --- | --- |
| auto | 有 CUDA GPU 时使用 GPU，否则使用 CPU |
| cuda | 强制使用 CUDA |
| cpu | 强制使用 CPU |

### 7.2 预览转码配置

```env
CARMATE_PREVIEW_MAX_WIDTH=1280
CARMATE_PREVIEW_CRF=23
CARMATE_PREVIEW_TRANSCODE_TIMEOUT_SECONDS=300
```

如果系统 FFmpeg 不在 PATH 中，可以配置：

```env
CARMATE_FFMPEG_PATH=D:\tools\ffmpeg\bin\ffmpeg.exe
```

如果未配置，后端会尝试使用 `imageio-ffmpeg`。

## 8. 当前实现的优点

- 支持 8 种标准交警手势以及“无手势”类别。
- 使用人体关键点和 LSTM 时序模型，符合连续动作识别需求。
- 上传视频支持边分析边返回，前端不需要等完整视频分析结束才显示结果。
- 前端支持浏览器兼容预览转码，避免视频编码不兼容导致无法播放。
- 视频播放过程中可以叠加当前动作标签。
- 支持动作时间线和动作分布，适合一个视频中包含多段动作的情况。
- 支持实时摄像头画面识别，可以用于实时演示。
- 支持 GPU 推理，可提升视频分析速度。

## 9. 当前局限与后续优化

当前实现仍有以下局限：

- 实时摄像头是定时截帧上传，不是严格意义上的连续视频流推送。
- 当前实时采样间隔约为 `1200ms`，动作很快时可能漏检。
- LSTM 需要一定连续帧积累，因此动作标签可能会略有延迟。
- 视频识别准确率依赖姿态估计质量，遮挡、距离过远、画面模糊会影响结果。
- 浏览器预览转码会增加一次后端处理开销，长视频可能需要等待更久才能得到兼容 MP4。

后续可优化方向：

- 将实时摄像头采样间隔调整到 `500ms-800ms`。
- 对实时结果加入滑动窗口平滑，减少标签跳变。
- 使用 WebSocket 传输实时帧和结果，降低请求开销。
- 对长视频增加后台任务和任务进度查询。
- 增加关键点可视化，在视频上显示人体骨架。
- 增加更多测试视频，统计每类手势准确率和混淆矩阵。

从课程要求角度看，本模块已经具备：

- 识别 8 种中国标准交警指挥手势的能力。
- 基于人体关键点和 LSTM 时序特征的动作分类流程。
- 输出手势类别和置信度。
- 支持上传视频和实时摄像头画面处理。
- Web 端展示识别标签、结果摘要和视频动作时间线。

整体上，该模块已经形成了从模型推理到前端展示的完整闭环，可用于项目演示和功能验收。
